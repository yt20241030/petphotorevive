## 2. 关键技巧:双引擎抽象(务必这样搭)

把"修图"抽象成一个统一接口,背后两个实现:

- `sharp-basic`(**临时引擎,不需要任何密钥**):用 `sharp` 做基础放大 + 锐化,让上传的老照片**确实有可见的清晰化提升**。现在就用它把全流程跑通、上线。
- `replicate-realesrgan`(**真引擎,需 token**):调 Replicate `nightmareai/real-esrgan`,`face_enhance=false`、`scale=2`。

**切换逻辑:检测到环境变量 `REPLICATE_API_TOKEN` 存在,就自动用真引擎;否则用临时引擎。** 这样 token 一贴即切真,产品结构不动。

## 3. 本阶段任务清单(按顺序做,每步能跑/能测)

**A. 完整产品落地页(英文、单页、响应式)**
- 内容:简短有力的价值主张、上传控件、**前后对比滑块(hero)**、定价与"下载高清"按钮、页脚 "by hipopo"。
- 桌面优先,但**必须在手机浏览器能正常打开、上传、操作**(Next.js 默认响应式即可,别做手机专属界面)。
- 语气:温暖、可信、克制(面向珍视宠物的人群)。

**B. 上传 → 后端接收 + 校验**
- 前端上传控件 → 服务端接收;校验:仅 jpg/png/webp、单张 ≤ ~10MB、拒绝超大图。

**C. 修复管道(双引擎,见第 2 节)**
- 现在走 `sharp-basic` 临时引擎跑通;结构上预留 `replicate-realesrgan`,由 `REPLICATE_API_TOKEN` 自动切换。

**D. 水印 + 降分辨率预览**
- 用 `sharp` 生成:①打水印("PetPhotoRevive")、压分辨率的**预览图**用于展示;②干净高清原图存服务端私有位置,按 `jobId` 归档,**绝不暴露给前端**。

**E. 付款门 + 高清释放**
- 页面展示前后对比(滑块)+ 带水印预览 + 按钮 `Download HD (no watermark) — $6`(价格做成配置项)。
- 干净高清**仅在付款成功后**通过一次性/带签名链接释放。

**F. Dodo 收银 + Webhook（测试模式）**
- 若你能拿到 Dodo 的 **test 模式密钥**,现在就用 test mode 接入 Checkout + Webhook,**验签**后才放行下载。
- 拿不到 test key 就先把这块脚手架搭好、留占位,等创始人提供。
- 严禁凭前端跳转就放行——必须以服务端验签的 Webhook 事件为准。

**G. 成本闸(防刷)**
